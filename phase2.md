# EcoLens Phase 2: Ecosystem Similarity Retrieval Framework

This document outlines the architectural additions, objectives, and specific implementations introduced in Phase 2 (Objective 2) of the EcoLens project.

---

## 1. What We Added

To build a formal retrieval engine on top of the Phase 1 embeddings, we introduced the following components to the repository:

1. **`06_retrieval_engine.py`**: The core search engine module.
2. **`07_evaluate_retrieval.py`**: The evaluation and metrics module.
3. **`08_retrieval_dashboard.py`**: The visualization and interactive UI module.
4. **Configuration Updates**: Added `RESULTS_DIR`, `DEFAULT_TOP_K` (10), and `EVALUATION_K_VALUES` ([1, 3, 5, 10]) to `config.py`.
5. **Output Artifacts**: Introduced a `results/` directory to store `retrieval_results.json`, `analog_database.json`, and `evaluation_report.json`.

## 2. What We Are Doing & How We Are Doing It (Code Breakdown)

### Step 6: The Retrieval Engine (`06_retrieval_engine.py`)

**What we are doing:**
We are building an engine that takes a "query" ecosystem patch and searches the entire database to find the mathematically closest ecological analogs using three different similarity measures.

**How we are doing it:**
1. **Loading Data**: The script loads the 768-dimensional embedding vectors (generated by Prithvi-100M in Phase 1) from the `embeddings/` directory and reads the `metadata/catalog.json`.
2. **Leave-One-Out Searching**: It loops through every single patch in the database. For each patch (acting as the "query"), it compares its embedding vector against all other patches in the database.
3. **Similarity Math**: The comparison is done using three strict mathematical formulas:
   - **Cosine Similarity**: Uses `numpy.dot()` and vector norms to measure the angle between two vectors. Returns a score up to 1.0.
   - **Euclidean Distance**: Uses `numpy.linalg.norm(a - b)` to measure the straight-line physical distance between vectors, which is then converted into a similarity score via `1 / (1 + distance)`.
   - **k-Nearest Neighbor (kNN)**: Finds the K closest patches by raw Euclidean distance, sorted in ascending order.
4. **Ranking & Saving**: For each method, the results are sorted from most similar to least similar. The complete ranked lists for all methods are saved to `results/retrieval_results.json`. Additionally, the top-10 results for each patch (using cosine similarity) are extracted and saved as the final `results/analog_database.json`.

### Step 7: Retrieval Evaluation (`07_evaluate_retrieval.py`)

**What we are doing:**
We are grading the retrieval engine's accuracy using standard Information Retrieval (IR) metrics to see how well the AI model groups similar ecosystems together, and identifying where it gets confused.

**How we are doing it:**
1. **Defining "Relevant" Ground Truth**: The script establishes a strict rule: If a query patch is a `forest`, then only other patches labeled `forest` are considered "relevant" (correct) search results.
2. **Computing Standard IR Metrics**:
   - **Precision@K (P@K)**: Looks at the top-K retrieved items and calculates the percentage that are relevant (e.g., if 4 out of the top 5 results are the same ecosystem, P@5 = 0.80).
   - **Recall@K (R@K)**: Calculates what percentage of the *total possible* relevant patches were successfully found in the top-K results.
   - **Mean Reciprocal Rank (MRR)**: Looks at the rank of the *first* correct result. If the first result is correct, the score is 1.0. If the first correct result is at rank 3, the score is 1/3 (0.33).
   - **Mean Average Precision (mAP)**: Averages the precision scores at every rank position where a relevant item is found, providing a single overall quality score.
3. **Per-Category Analysis**: The script groups the queries by their ecosystem category (Forest, Wetland, Mangrove, Agricultural, Urban Green) and calculates the mAP and P@K for each category independently to see which ecosystems the model understands best.
4. **Confusion Matrix**: It tracks every time the engine retrieves an incorrect category and calculates the proportion of times, for example, a Mangrove query incorrectly returns an Urban Green patch.
5. **Output**: The script saves all calculated metrics into `results/evaluation_report.json` and prints comparative tables to the console.

### Step 8: The Interactive Dashboard (`08_retrieval_dashboard.py`)

**What we are doing:**
We are generating a local, interactive web application to visually explore the similarity search results, the evaluation metrics, and the multi-dimensional embedding space without needing a complex backend server.

**How we are doing it:**
1. **Dimensionality Reduction (PCA)**: The script takes the 768-dimensional embeddings and uses Principal Component Analysis (`numpy.cov` and `numpy.linalg.eigh`) to compress them down to 2 dimensions (PC1 and PC2) so they can be plotted on a flat X/Y scatter plot.
2. **Data Injecting**: The script reads the original metadata, the raw similarity calculations (Cos, Euc, kNN), and the evaluation metrics from `evaluation_report.json`. It converts these Python dictionaries into JSON strings.
3. **Standalone HTML Generation**: It creates a single HTML file (`retrieval_dashboard.html`). Using Python f-strings, it injects the JSON data directly into a `<script>` tag inside the HTML file. 
4. **Frontend Implementation**: 
   - Uses `Chart.js` to render the 2D scatter plot and the per-category mAP bar chart.
   - Uses vanilla JavaScript to build an interactive UI: when a user clicks a point on the scatter plot or selects a patch from the dropdown, a function filters the injected JSON data to instantly display the top-15 most similar patches for the currently selected similarity method.
   - Dynamically generates an HTML table to render the Confusion Matrix, applying conditional background colors to create a heatmap effect based on the confusion proportions.
